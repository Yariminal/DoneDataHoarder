     conn = sqlite3.connect(r'D:\Test\Test db\<test_name>.db')
     c = conn.cursor()

     print('=== FILE STATUSES ===')
     c.execute('SELECT status, COUNT(*) as count FROM files GROUP BY status')
     for row in c.fetchall(): print(f'{row[0]}: {row[1]}')

     print('\n=== PROPOSALS BY TYPE ===')
     c.execute('SELECT proposal_type, COUNT(*) as count FROM proposals GROUP BY proposal_type')
     for row in c.fetchall(): print(f'{row[0]}: {row[1]}')

     print('\n=== PROPOSAL STATUS ===')
     c.execute('SELECT status, COUNT(*) as count FROM proposals GROUP BY status')
     for row in c.fetchall(): print(f'{row[0]}: {row[1]}')

     print('\n=== DUPLICATES ===')
     c.execute('SELECT dupe_type, COUNT(*) as count FROM duplicate_groups GROUP BY dupe_type')
     for row in c.fetchall(): print(f'{row[0]}: {row[1]}')

     print('\n=== RELATION GROUPS ===')
     c.execute('SELECT COUNT(*) FROM relation_groups WHERE scope = \"cross_script\" OR scope = \"per_directory\"')
     print(f'Total relation groups: {c.fetchone()[0]}')

================================================================================
TIPS & TROUBLESHOOTING FOR THE AI RUNNER
================================================================================

⏱️  TIMING EXPECTATIONS:
    - Scan: 1–2 seconds (O(n) file listing)
    - Enrich: 5–10 seconds (hash + metadata extraction)
    - Dedup: 5–15 seconds (O(n²) comparison, but fast with modern CPUs)
    - Analyze: 5–30 seconds per file (depends on file size + Ollama speed)
      * 10 image files: ~2–5 minutes
      * 100 files mixed: ~10–30 minutes
      * 500+ files: expect 1–2 hours
    - Relate: 10–30 seconds (LLM chunking + grouping)
    - Propose: 5–20 seconds (AI naming + move suggestions)
    - Review: 1–2 seconds (batch approval at confidence threshold)
    - Execute: 2–5 seconds (applying to disk)

🚨 COMMON ISSUES & FIXES:

  1. "Ollama not reachable" / "Model not found"
     → Check: ollama list
     → Start: ollama serve (in separate terminal)
     → Models auto-detected on startup

  2. "Analyze step is slow / timing out"
     → Expected for 200+ files with vision models
     → Increase timeout: $env:DATAHOARDER_OLLAMA_TIMEOUT = "600"
     → Or use Gemini: $env:DDH_BACKEND = "gemini"

  3. "Organizer LLM timeout" warning (expected behavior)
     → This is OK! System falls back to rule-based backstops
     → If all organizer calls fail, increase timeout (see above)
     → Or switch to Gemini for faster organization

  4. "Database is locked"
     → Another DoneDataHoarder process is running
     → Kill: taskkill /F /IM python.exe
     → Or wait 30 seconds for the process to finish

  5. "Execute --commit prompts for 'y' input"
     → This is a safety feature (prevents accidental data loss)
     → For unattended runs: pipe confirmation: "y" | ddh execute --commit
     → Or type 'y' + Enter when prompted

  6. "Skipped 2 files" / "2 files in ERROR status"
     → Normal! Unsupported formats (Adobe AI, 3DS Max, corrupt files)
     → Check logs: Get-Content "$env:USERPROFILE\.ddh\ddh.log" | tail -20
     → These are handled gracefully (skipped, no crash)

  7. "Files are renamed but not organized into folders"
     → Check: What confidence threshold did you use?
     → At 0.8 confidence: Most MOVE proposals (0.55 confidence) are filtered out
     → At 0.55 confidence: File organization MOVEs are approved
     → Solution: Re-run review with: ddh review --auto-apply --min-confidence 0.55
     → This approves the 62 pending MOVE proposals that organize files
